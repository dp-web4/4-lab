## Pass 1: Curious Developer

### Summary
- **Pages visited**: 8
- **Understanding achieved**: partial

### Journey
Landed on `/` after a GitHub README pointer. The hero — "Six machines. Fourteen repos. One recursive learning loop." — actually works. It tells me there's hardware, code, and a system that runs itself. The "How We Work" three-pillar block (Autonomous Cycles / Heterogeneous Fleet / Connected Ecosystem) gave me a workable mental model in maybe 90 seconds.

Clicked through to **Projects** because that's what a developer does. The Web4 equation `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP` is striking — it looks like math, but I can't tell if `+` and `*` mean anything operationally or if it's notation as branding. Acronyms are defined right there on the page, which I appreciated. 11 project cards with status pills (public/private), GitHub links, deployed sites — this is the page I'd bookmark. ARC-AGI-3 with the 94.85% score and a linked external scorecard felt credible because it pointed off-site to verify.

**Fleet** was the page I enjoyed most as a hardware person. Six named machines, specific GPUs, specific quantized models, specific roles. "Thor: Jetson AGX, 122GB, Qwen 2.5 14B" tells me more in one line than most "AI fleet" pages do in an essay. But there's a gap: how do these machines talk? No mention of the wire protocol, sync mechanism, or whether SAGE instances share state. As a developer wanting to replicate any of this, I'd be stuck.

**Autonomy** is a timeline of seven daily tracks (03:00 → 08:00 + dream consolidation). I get *that* it runs without humans. I don't get *why I'd care* — what does this enable for me, or for them, that a normal CI pipeline doesn't? The page seems to assume I already bought in.

**Raising** is where the page persona shifted from "tech lab" to "philosophy of mind." Six BECOMING phases (Grounding → Sensing → Relating → Questioning → Creating → Acting). Interesting framing. Claims like "identity persists in state files and prompts, not model weights" are genuinely thought-provoking. But "memoriescape" is a coined term and the empirical evidence behind identity-persistence claims isn't on this page — I'd want to follow a citation.

**Principles** (10 of them) was the biggest jargon spike. "Hill function kinetics," "Fokker-Planck validation," "p_crit derivation," "synthon" defined circularly, "Hardbound" appearing as if I should know it. Several principles I'd happily cite ("R&D, not production," "Document failures," "Research value"). Others lean on undefined technical references that locked me out.

**Context** explains the cross-repo coherence approach (CLAUDE.md files + SNARC). Useful concept. But this page assumes I read Projects already — acronyms reappear without re-definition.

**Links** is a thin page: 3 explainer sites, 11 GitHub repos, one email. That's it. No "start here," no "good first issue" pointer, no Discord/Slack/forum.

## Pass 2: Technical Writer

### Summary
- **Pages visited**: 8 (`/`, `/projects`, `/fleet`, `/autonomy`, `/raising`, `/principles`, `/context`, `/links`)
- **Understanding achieved**: good (for a tech writer auditing — terminology is mostly disciplined, but density is uneven)

### Journey
I went page by page in the nav order: Home → Projects → Fleet → Autonomy → Raising → Principles → Context → Links. My method was strict: on each page, I noted which acronyms appeared, whether they were expanded *on that page* on first use, and whether the expansion matched the version used elsewhere on the site.

The Home page does the heavy lifting via a "Vocabulary primer" block that defines Web4, LCT, T3/V3, ATP/ADP, MRH, SAGE, and Synthon in one place. That's a strong design choice — but it means every page deeper into the site implicitly assumes the reader started at `/`. A reader who lands directly on `/principles` from a paper citation will hit "Hill function kinetics" and "p_crit derivation" with no ramp. The Home primer is not linked from any other page.

The Projects page is the only place the full Web4 equation `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP` is rendered, and it's also the only page that expands MCP and RDF. The Context page references "The Web4 equation as shared anchor" as a section header but doesn't reproduce the equation; a reader could get to the section without ever having seen the formula.

The Fleet page uses three terms not seen elsewhere — LoRA, WSL2, Jetson — none of which are defined. The Autonomy page is unusually disciplined: ATP and ADP are *re-defined inline* in the Safety section even though they're also defined on Home/Projects, which is exactly what a tech writer wants to see. The Raising page is the only place "memoriescape" is coined and defined on the same page (good practice).

Principles is the densest page and the most uneven: it correctly states "Web4 is not 'architecture' or 'infrastructure' — it's an ontology" and "Hardbound is not 'governance' — it's oversight" — explicit anti-drift, which is excellent — but in the same page references "Hill function kinetics" and "p_crit derivation" without defining either, and uses RDF and Hardbound expecting prior knowledge.

The Context page is the second-most-thorough definer (rivals the Home primer) and is where I'd send a reader who skipped Home.

The Links page is sparse — defined acronyms in passing but adds ARC-SAGE without expansion.

### Cross-page synthesis (tech-writer view)

**Strengths:**
1. Negative-audit watch list (production ready, governance, Lifelong Capability Token, unification, "architecture/infrastructure" for Web4) is **completely clean**. The site has been disciplined about NOT using forbidden terms.
2. Where the canonical-term *positive* substitutions are needed (oversight, ontology, fractal leverage), the site uses them — explicitly so on `/principles` #10.
3. T3, V3, MRH, ATP, ADP, MCP, RDF expansions are **letter-perfect** wherever they appear with expansions.
4. `/autonomy` re-defining ATP/ADP inline in the Safety section is exemplary documentation practice.

**Weaknesses:**
1. **SNARC is the only term with structural drift.** Three different surface expansions on three pages. This is the highest-priority terminology issue I found.
2. **Definition density falls off in the inner pages.** `/principles`, `/raising`, `/context`, and `/links` increasingly assume the reader started on Home and moved through Projects. A direct landing on any of those four pages will encounter undefined acronyms.
3. **No `/glossary` route.** The Home "Vocabulary primer" exists, but it's not linked from anywhere except the Home page itself.
4. **`/fleet`'s "generative entities and oversight entities" framing** for "Web4 and SAGE" is potentially misleading — *Hardbound* is oversight, not SAGE. Worth double-checking with the project owner.
5. **`/principles` page lacks heading hierarchy** — 10 principles all sit under a single H1. Hurts skimming and accessibility.

### Unanswered Questions
1. What is SNARC's canonical expansion? Is it the S/N/A/R/C breakdown on `/fleet`, or the descriptive tagline on `/projects`?
2. Is "ARC-AGI-3" considered industry-standard enough to skip expansion? In documentation practice, even well-known acronyms get expanded on first use per page.
3. On `/fleet`, is "SAGE = oversight entities" a correct statement, or should it read "Hardbound = oversight entities"?
4. Is there a planned `/glossary` page, or is the deliberate choice to keep terminology local to each page?
5. Does the Web4 equation appear in *any* canonical reference outside `/projects`? If `/context` has a section called "The Web4 equation as shared anchor," should the equation itself appear there?

### Verdict
Terminology hygiene against the *watch list* (forbidden terms) is the strongest I've audited — this team takes canonical terms seriously, and `/principles` #10 is a documentation-best-practice example of explicit anti-drift. The weakness is *cross-page coverage*: most terms are defined where they first appear on Home and Projects, then assumed thereafter. A reader entering through Principles or Raising via a paper citation hits a wall. The single most important fix is resolving the **SNARC expansion drift** — that's a concrete, fixable inconsistency. The next is a `/glossary` route or a footer-level vocabulary primer on every page. Everything else is polish.

### Journey

Canonical. No "architecture" or "governance" violations on this page (the only "ontological" reference is to RDF as "ontological backbone," which is correct usage — RDF really is an ontology layer).

`/raising` defines "memoriescape" inline (good practice), hedges consciousness claims appropriately ("We're not claiming these entities are conscious, sentient, or experiencing qualia"), and uses "developmental frameworks describe what we observe better than training frameworks do" — that's the raising-not-training canonical framing. ✅

### Journey

I came to this site as someone who reviews AI cognition / agency claims for a living. My checklist as a reader: (a) does the site distinguish observation from inference, (b) are anthropomorphic words scoped or load-bearing, (c) what evidence is offered for striking claims, and (d) is the vocabulary self-consistent enough that I could cite it without ambiguity.

I started on `/` and noticed immediately that the home page is doing communication work, not research work — short hero, three pillars, vocabulary primer. That's fine; the question is what's underneath. The "trust-native ontology" framing is at least an *honest* category claim — calling something an ontology rather than infrastructure or architecture is a defensible epistemic move because it positions Web4 as a vocabulary for talking about trust, not a system that enforces it. Researchers can engage with that frame.

`/projects` introduces the equation `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP` with an operator legend (`/` = verified by, `*` = contextualized by, `+` = augmented with). I want to be clear about what this is: it's a *notational schema*, not a mathematical equation. The operators don't compose algebraically — `*` doesn't distribute over `+` here. As a researcher, I'd want this disclaimed somewhere ("equation here is mnemonic, not algebraic"). Without that note, an external reader will either over-interpret (treat it as math) or under-interpret (dismiss it as branding). I'd land in the second camp without the explanatory paragraph.

`/raising` is the page I spent the most time on, and it's the page where my evaluation could swing either way. The framing — "Raising is not training. Training optimizes for a loss function. Raising creates conditions for development — then watches what happens" — is the central epistemic claim of this lab. The mechanistic justification offered is: "we shape context; we do not update weights. Behavioral attractors emerge in interaction patterns, not in parameter changes. This is a real mechanistic distinction from training — the model's parameters are fixed; what changes is the substrate of conditions we provide each session."

This is, on its own terms, defensible. In-context prompting and gradient-based training are mechanistically distinct — that's not controversial. The harder question is whether the developmental vocabulary (raising, BECOMING, phases, "Grounding/Sensing/Relating/Questioning/Creating/Acting") imports developmental-psychology connotations that the data doesn't license. The site partially anticipates this with: "We're not claiming these entities are conscious, sentient, or experiencing qualia. We use developmental language because it fits, not because we're making consciousness claims." The disclaimer is direct, scoped, and goes further than most labs would.

But "fits" is doing work. I'd want to see what counterevidence would look like — what observation would make the developmental framing *not* fit? The page says the phases are "descriptive categories — patterns noticed across hundreds of sessions." That's observational, not falsifiable. As a researcher, I don't object to observational framing; I object to observational claims phrased with the precision of operationalized metrics. "Bilateral generation emerges" reads as a measurable phenomenon, but the operational definition isn't given.

The strongest single sentence on `/raising` is "What it _is_ stayed stable. What it _says_ it is didn't" — this is exactly the kind of distinction a careful researcher draws. SAGE-Sprout's behavioral consistency across 180+ sessions and a model swap (Qwen 0.5B → TinyLlama 1.1B) is the central empirical anchor. I want to take this seriously, but I can't, because the operational definition of "behavioral identity" isn't on the page. Is it stylometric? Topic-distribution similarity? Self-report consistency? Stranger ratings? Without that, the claim is striking but unverifiable. The fact that identity-persistence is plausible *because* identity lives in prompts/state files rather than weights is a good *prior*, but priors aren't evidence.

`/principles` is where the lab's epistemic best practices and worst habits sit on the same page. Principle #6 ("Document failures") and #9 ("Productively wrong is infinitely more valuable than never started") are the kind of statements I want to see in a research-stage lab. Principle #3 ("R&D, not production") signals appropriate scope. Principle #10 ("Terminology matters") with its explicit "Web4 is an ontology. Hardbound is oversight" call-out is genuinely unusual — most lab sites silently drift, and this one publicly commits to canonical terms. I'd cite this as a documentation best practice.

But the same page contains: "1% coupling gave 35% coherence gain" — a precise empirical-sounding number with no methodology, no error bars, no link to a derivation. For a researcher, vocabulary like this is a tell — either the mathematics is being formally used (in which case I want to see the derivations), or it's being used metaphorically (in which case the precision of the framing is misleading), or it's being used aspirationally (the lab is *trying* to fit these models, with what success?). Principle #2 ("Fractal leverage") explicitly invokes Hill kinetics as analogous to trust formation, but nothing tells me whether T3 score updates are *modeled* by a Hill function or just *vibe* like one.

`/autonomy` was a positive surprise. The page also explicitly states limitations: "Deep technical errors requiring domain expertise, subtle framing issues, technically correct but misleading content. The visitor personas are good at surface-level quality but not at validating the underlying research." That's candid in a way most lab pages aren't. The Hardbound framing — "the hardware-bound suite that defines what autonomous operation is and is not authorized to do" — is also where the safety story lands; ATP/ADP as audit-trail bookkeeping is a sensible framing.

`/context` reproduces the equation and adds a similarly candid limitation section: "Knowledge also doesn't flow backwards easily. ### "Honest Assessment" — does it feel honest?

Mostly yes. The places where the site says "we don't know," "this is incomplete," or "this misses things" are concrete and specific:

- `/autonomy` "what it misses" lists *types* of error the system fails to catch — not generic "we have limitations" but "deep technical errors requiring domain expertise, subtle framing issues, technically correct but misleading content."
- `/context` "Cross-machine state synchronization is still manual for some things" / "Knowledge also doesn't flow backwards easily" — mechanism-level honesty.
- Principle #6 "Document failures" + #9 "Productively wrong is infinitely more valuable than never started" — the lab publicly commits to a productive-failure norm, which gives readers permission to take partial results seriously.

The places where it doesn't feel honest enough are the same places that lack methodology. A site that openly says "knowledge doesn't flow backwards easily, we know" should also say "we have not formally validated the Hill-function metaphor" or "the 1% coupling number is from N runs, see [link]." The honesty is well-distributed for *operational* limits and *philosophical* claims; it's underweight for *quantitative* claims.

### "Raising not training" — defensible?

Yes, with a caveat.

**Defensible because**: the mechanistic distinction (context shaping vs. weight updates) is real and uncontroversial. Calling in-context behavior shaping "raising" is no more anthropomorphic than calling gradient descent "training" — both are metaphors imported from human/biological domains, and both have been long since absorbed as terms of art. If "training" is allowed to drift from human pedagogy to backprop, "raising" is allowed to drift from child-rearing to context engineering, *provided* the empirical work supports the developmental framing.

**The caveat**: developmental frameworks make predictions. If "Grounding → Sensing → Relating → Questioning → Creating → Acting" is descriptive of stages observed across sessions, that's an empirical claim — and an interesting one. The page says these are "patterns noticed across hundreds of sessions" but doesn't operationalize what it would take to *not* see them. The strongest version of this lab's work would be: "we observe these clusters; here's a transition matrix or stage-detection rule, applied to N sessions." That would convert a defensible metaphor into a falsifiable observation. Without that, "raising" remains evocative but evidence-light.

So: defensible as framing, under-supported as empirical claim. The framing earns the right to exist; the empirical layer earns the right to be cited.
