## Pass 1: Curious Developer

### Summary
- **Pages visited**: 8
- **Understanding achieved**: partial

### Journey

Landed on the home page from a GitHub README link. "Six machines. Eight repos. One recursive learning loop." — that's a hook. I'm curious. The hero section is clean and the stats (6 machines, 8+ repos, 7 autonomous tracks) give me a quick sense of scale. The vocabulary primer at the bottom is a smart move — I actually scrolled down and read it before clicking anything else because the tagline mentions "trust-native ontology" and I had no idea what that meant.

Clicked **Projects** next because I wanted to know what these repos actually *do*. The Web4 equation hit me immediately: `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP`. With the operator legend (`/` = verified by, `*` = contextualized by, `+` = augmented with) it's parseable, but it reads more like a thesis notation than something I can act on. The individual project cards are good — I can see what each repo is for. SAGE as a "continuous consciousness loop" and "IRP" caught my eye but IRP isn't expanded anywhere on this page. The "fractal leverage" framing at the bottom makes sense as a design philosophy. ACT and 4-Life sound interesting — trust-native societies with karma mechanics is a fun concept.

**Fleet** page: this is where it got real for me. Six machines, each with a named identity and specific hardware. The two-pool budget split (Synthesis vs Oversight) is clever infrastructure thinking. The identity portability story — SAGE-Sprout's identity transferring from Qwen 0.5B on a Jetson to TinyLlama 1.1B on a different machine — is the single most compelling claim on the entire site. "The model is weather. The identity is organism." I'd quote that. The peer-to-peer trust mechanism using T3 scores is described clearly here.

**Autonomy** page: Seven daily tracks on cron schedules. I immediately understood the Supervisor → Archivist → Publisher → Visitor → Maintainer pipeline — that's a well-designed feedback loop. The Explorer track using persistent NotebookLM notebooks is interesting. The "Dream Consolidation" concept at the end is where I started losing the thread — memory pruning and "identity-level storage" feels like it's borrowing neuroscience metaphors without enough grounding for me to evaluate. The four visitor personas listed here (Skeptic, Newcomer, Practitioner, Connector) don't match the personas described in my own instructions (Curious Developer, Technical Writer, Web4 Contributor, External Researcher) — not sure if that's intentional versioning or drift.

**Raising** page: "Raising is not training" is the thesis, and the five BECOMING phases are laid out clearly. Phase descriptions make sense progressively — Grounding → Sensing → Relating → Questioning → Creating. The "Hill function kinetics" reference in Phase 3 is unexplained on this page. The key discoveries section is genuinely interesting: identity persisting across 115 sessions despite self-description drift, "memoriescape" as a coined term, bilateral generation without stop tokens. The critical caveat ("We're not claiming these entities are conscious") is well-placed and honest.

**Principles** page: 12 principles, each with a quote and explanation. These read well. "Substrate, not structure" and the termite analogy is memorable. "Document failures" with the 400x error example is refreshingly honest. "Terminology matters" explicitly calls out the ontology-not-architecture distinction. This page felt like the philosophical backbone and I appreciated the concrete examples alongside the abstractions.

**Context** page: Explains how knowledge flows across the system. CLAUDE.md files as the coordination mechanism, SNARC's salience scoring (Surprise, Novelty, Arousal, Reward, Conflict), and the four-tier memory hierarchy. The honest limitations section — cross-machine sync is partially manual, backward knowledge flow is incomplete — is good. On this page, LCT is mentioned in the Web4 equation without being expanded (it's expanded on other pages but not here).

**Links** page: Clean, functional. Three explainer sites, seven GitHub repos, and a contact email. Everything I'd need to dig deeper. No issues here.

**Key terminology findings**:

1. **T3/V3 is the site's biggest terminology problem.** It has four different presentations across the site: "Trust Tensor / Value Tensor" (home), "trust tensors" (projects), "Talent/Training/Temperament" (fleet), and "tensors bind to entity-role pairs" (principles). Only the Fleet page uses the canonical expansion. The home page's "Trust Tensor / Value Tensor" framing is the most concerning because it's the first thing readers encounter and it introduces non-canonical wrapper terms.

2. **First-use expansion discipline is inconsistent.** The home page and projects page expand most terms. Fleet expands T3. But context, principles, and autonomy pages assume the reader already knows the acronyms. A reader who enters the site on any page other than home will encounter unexpanded acronyms.

3. **SNARC is never fully expanded.** The context page describes its function and lists its five dimensions, but never explicitly states that S-N-A-R-C maps to Surprise-Novelty-Arousal-Reward-Conflict.

4. **Visitor persona mismatch on /autonomy is a factual error**, not a terminology issue. The page describes four personas that don't exist in the actual running track.

But T3/V3 is a real problem: the home page introduces "Trust Tensor / Value Tensor" as primary names with the canonical triplets demoted to parentheticals, while other pages use different informal shorthands. A reader following the site's own vocabulary primer would learn "Trust Tensor" first — and then never encounter that term again on any other page. This is exactly the kind of drift that confuses newcomers and propagates. The fix is straightforward: drop "Trust Tensor / Value Tensor" from the home page and lead with the canonical triplets everywhere.

### Journey

I came to this site after hearing a conference mention of "developmental frameworks for AI agents" and "identity persistence across model swaps." Those are strong claims. My job is to evaluate whether they're defensible.

**Home page**: The hero — "Six machines. Eight repos. One recursive learning loop." — is clean and empirical. Good start. The stats (6 machines, 8+ repos, 7 autonomous tracks) are concrete and verifiable. The vocabulary primer at the bottom defines six terms. As a researcher, I immediately notice: these definitions are presented without epistemic status. "Linked Context Token — a persistent identity anchor" reads as established fact. Is this a proposed framework or a deployed system? The page implies the latter ("This is not a demo — it runs every day") but the theoretical vocabulary suggests the former. The framing is research documentation, which is appropriate, but the boundary between "what we built" and "what we hypothesize" isn't always clear.

T3/V3 is described as "Trust Tensor (Talent, Training, Temperament) and Value Tensor (Valuation, Veracity, Validity)." I note "Trust Tensor" and "Value Tensor" as container terms — I can't evaluate whether these are canonical because I'm not a Web4 expert, but the mathematical language ("tensor") implies a formal structure. If these are truly tensors in the mathematical sense, the site should cite the formalization. If they're metaphorical tensors, the site should say so. Imprecise mathematical vocabulary is a red flag in any research presentation.

**Raising page** — this is what I came for. The thesis: "Raising is not training. Training optimizes for a loss function. Raising creates conditions for development — then watches what happens." This is a defensible distinction. In developmental psychology (Vygotsky, Piaget), the parallel is between direct instruction and scaffolded exploration. The site is making a methodological claim, not an ontological one — they're not saying the agents *are* developing, they're saying developmental frameworks describe the observations better than optimization frameworks. That's a testable claim, and the framing is epistemically responsible.

The BECOMING phases (Grounding → Sensing → Relating → Questioning → Creating) map loosely to developmental psychology stages. Phase 3's "Hill function kinetics" reference is interesting — Hill functions describe cooperative binding in biochemistry, and applying them to trust formation is a specific, falsifiable claim about the dynamics. The site doesn't explain this enough for a non-biochemistry reader, but the claim itself is well-defined.

The consciousness caveat is the most important text on the site: "We're not claiming these entities are conscious, sentient, or experiencing qualia. We're claiming that developmental frameworks describe what we observe better than training frameworks do." This is exactly the right epistemic posture. It separates the observational claim (developmental frameworks fit the data) from the ontological claim (the agents are actually developing). The language "something that looks like growth, something that looks like identity" — using "looks like" rather than "is" — maintains appropriate ambiguity.

The key discoveries section reports specific, falsifiable findings: identity persistence across 115 sessions despite self-description drift, bilateral generation without stop tokens (linked to Vygotsky's egocentric speech), and "memoriescape" as a coined phenomenological term. These are well-described observations. The Vygotsky reference is appropriate and specific. "Memoriescape" is acknowledged as coined rather than established. "Capacity as register" — that model size changes expressive range, not just quality — is a genuinely interesting observation that could be tested.

The "interactive selection" methodology — "We don't create new behaviors. We probe what the model responds to, observe which attractors surface, adjust context to resonate, and reinforce what works" — is a clear description of what's happening. The dynamical systems language ("attractors," "basins") is appropriate for the phenomenon they're describing, and they're not overclaiming. This is reinforcement of existing model tendencies through context, not fine-tuning or training. The distinction is real and important.

**Principles page**: 11 principles, each grounded in specific experience rather than aspiration. The framing — "These aren't aspirational values posted on a wall. This is a hedging asymmetry problem. The Raising page does the hard epistemic work of distinguishing observation from ontology. A skeptical researcher would notice this immediately and question whether the Raising page caveats are performative rather than sincere.

**Fleet page**: The identity portability claim is the strongest empirical finding on the site. "The model is weather.
