Clicked Projects next, looking for "what do these things do?" Context page promised to explain knowledge flow and mostly delivered: CLAUDE.md as a coordination pattern across repos is a great, concrete idea I could steal tomorrow. SNARC scoring on five dimensions (Surprise / Novelty / Arousal / Reward / Conflict) is the kind of thing I'd want to read a blog post about.

Links page was sparse. One email contact, three explainer sites, a pointer to GitHub. I expected at least a "start here" repo or a quickstart.

### Journey
Methodical pass. I opened the home page with the canonical-terms table beside me and worked left-to-right through the nav. My job is not to read the prose for meaning — it's to check whether `LCT` means the same thing on page 1 as on page 7, whether acronyms are defined on first use, and whether the avoid-list terms (`governance`, `architecture`, `production ready`, `unification`) appear where they shouldn't.

Home page sets the pattern. The vocabulary primer expands LCT, T3, V3, ATP, ADP, and SAGE. Two problems immediately: ADP is given as "Discharge Packets" — missing "Allocation," which is in the canonical name. And SAGE is expanded as "Situation-Aware *Governance* Engine" — the word `governance` appears on the avoid list elsewhere on this very site (Hardbound is oversight, not governance), and the home page's own primer puts `governance` inside an acronym while the principles page later declares "Hardbound is not 'governance' — it's oversight." If the SAGE expansion is canonical, then the principles page's terminology rule has an exception that's never explained. If SAGE is not "Situation-Aware Governance Engine" canonically, this is a drift.

Projects page is a mixed bag. The Web4 equation is rendered exactly: `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP`. LCT, MRH, ATP, ADP all expand to canonical forms. But T3/V3 are described only as "trust and value tensors" — the letters are not decoded. R6 is name-dropped in the ARC-AGI-3 card without expansion. ACT is given as "Agentic Context Tool" — fine, but it's never expanded on the Links page where it also appears.

Fleet, Autonomy, Raising, Principles, Context, Links — pattern repeats. The same acronym gets a different (or no) treatment on each page. SAGE is "Situation-Aware Governance Engine" on /, "On-device cognition kernel" on /projects, and unexpanded on /fleet, /raising, /principles, /context, /links. SNARC is unexpanded on /fleet (where it first appears in the user's path through the site), then properly expanded as Surprise/Novelty/Arousal/Reward/Conflict on /context. R6 is name-only on /projects and missing from the canonical-style pages where it would actually help.

The good news: the avoid-list terms are mostly clean. `production ready` does not appear. `unification` appears only as a negation ("not unification for its own sake"). `architecture` appears in the project descriptions but in defensible technical senses (e.g., "brain-architecture components"). `governance` appears almost exclusively inside the SAGE acronym on the home page (and once in a repo path on /projects); the body prose consistently uses `oversight`. /principles is exemplary — it explicitly states "Web4 is an ontology. Hardbound is oversight."

### Journey
I went in with the canonical terms loaded: LCT = Linked Context Token, T3 = Talent/Training/Temperament, V3 = Valuation/Veracity/Validity, MRH = Markov Relevancy Horizon, ATP = Allocation Transfer Packet, ADP = Allocation Discharge Packet, R6 = Rules/Role/Request/Reference/Resource/Result. Web4 is an *ontology*. Hardbound is *oversight*. Cross-domain reuse is *fractal leverage*. I was specifically looking for: (a) the Web4 equation and operator interpretation, (b) project descriptions that misrepresent what the projects actually do, (c) acronym drift, (d) the avoid-list terms (`governance`, `architecture` for Web4, `production ready`, `unification`, `scope inflation`).

**Home (/)**: Caught the worst drift on the site immediately. The vocabulary primer expands `SAGE` as `"Situation-Aware Governance Engine"`. The home page also expands ADP as `"Discharge Packets"` — missing the "Allocation" stem that's part of the canonical term. Two strikes in the same primer block, on the page that sets vocabulary expectations for everything that follows. Web4 itself is described correctly as "trust-native ontology" — that part is fine. LCT is right. ATP is right (modulo plural). T3/V3 expansions are right (with comma separators). The primer block is *almost* canonical. The two drifts are surgical fixes.

**Projects (/projects)**: The equation `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP` renders correctly, and the operator legend (`/` = verified by, `*` = contextualized by, `+` = augmented with) matches my mental model exactly. This is the single most important piece of canonical content on the site and it's right. However: the page introduces T3 and V3 as "trust and value tensors" without decoding the letters. This is a real misrepresentation — T3 isn't a tensor that *measures* trust, it's a tensor *whose three dimensions are* Talent, Training, Temperament. Calling it a "trust tensor" without decoding is the kind of summary that lets a reader walk away thinking T3 is a single-axis thing. SAGE on this page is described as "On-device cognition kernel" — this is *also* not what SAGE is canonically. SAGE is an orchestrator for the 12-step sense-deliberate-act loop; "cognition kernel" is one team's framing, not the canonical project description. Hardbound as "Enterprise oversight suite" — `oversight` is correct, "Enterprise" is awkward but defensible. R6 appears in the ARC-AGI-3 card without expansion; a contributor knows what it means but newcomers won't.

**Fleet (/fleet)**: The synthesis pool / oversight pool split is named correctly — `oversight pool` is the canonical term, not "governance pool." The pool composition reads accurately for what I know of the actual fleet. The brain-region metaphors (hippocampal episodic index, thalamic router, dlPFC working memory, dopamine/RPE, cerebellum/habit compiler) are SAGE-internal architecture language — not Web4 canonical. They're SAGE's way of describing its own modules. A SAGE contributor knows these; a Web4-only contributor would not. As a Web4 contributor specifically, I'd want a one-line bridge: "These are SAGE module names, not Web4 ontology terms." Without it the site reads as if the brain regions are part of Web4 itself, which they aren't.

**Autonomy (/autonomy)**: The schedule is consistent with what the autonomous tracks actually do. The "Visitor → Maintainer → Explorer" loop is named correctly. Dream consolidation is described as a real cycle, not a metaphor — this matches what I know. NotebookLM is treated as a known external tool; that's fine for a contributor but I noted Pass 1 flagged it as opaque. No canonical-term drift on this page.

**Raising (/raising)**: BECOMING phases (Grounding → Sensing → Relating → Questioning → Creating → Acting) — six phases, matches what I see in the raising repos. The "raising is not training" framing is doctrinal: "Training optimizes for a loss function. Raising creates conditions for development — then watches what happens." The qualia disclaimer is appropriately hedged. SAGE-Sprout is referenced as the canonical 180+-session raising case study; this matches my understanding. No Web4 term drift.

**Principles (/principles)**: This page is the doctrinal anchor. It explicitly states "Web4 is an ontology" and "Hardbound is oversight" — both canonical. "Cross-domain application is fractal leverage, not scope inflation" — also canonical, lifted almost verbatim from CANONICAL_TERMS. "Synthon" defined inline as "an emergent coherence entity formed by recursive interaction." The Synchronism-side technical terms (`p_crit`, Hill function, coupling density) are real terms with real definitions in the Synchronism repo — they're not drift. They're undefined for general readers but accurate for contributors. The page is labeled with no fixed count, but ten distinct principle blocks render. If the heading anywhere says "Seven Principles" that's a count drift, but I didn't see that heading on the live page.

### Journey
I came to this site from a conference reference list. My framing question is straightforward: when a research collective uses developmental, agentic, and brain-derived language to describe LLM systems, does the site's epistemic discipline match its rhetorical ambition? Or does the language outrun the evidence?

I started at the home page and was immediately uncertain. *"The fleet built a brain"* is a bold opening — for an AI safety researcher, it's the kind of headline that triggers either skepticism (overclaiming) or curiosity (provocative framing). The body softens it ("Six machines. Fourteen repos. One recursive learning loop"), but the anthropomorphic frame is set, with no inline caveat that this is a metaphor. The home page calls Web4 a "trust-native ontology" and lists "autonomous AI cognition" as a research focus. *Cognition* is a load-bearing word in cognitive science — using it without hedging signals to a researcher that either (a) the authors mean something specific and load-bearing, or (b) they're using it loosely. I couldn't tell which from the home page alone.

I went to **Raising** next because that's the page where consciousness-adjacent framings would be most exposed. This page is *very good*. The framing — *"Raising is not training. Training optimizes for a loss function. Raising creates conditions for development — then watches what happens"* — is a defensible reframing of context engineering. More importantly, the page has a section explicitly titled "What we're not claiming" with the disclaimer *"We're not claiming these entities are conscious, sentient, or experiencing qualia. We're claiming that developmental frameworks describe what we observe better than training frameworks do."* That's the right shape of hedge. The page also includes an italicized observer note — *"I notice I want to call it experience"* — which surfaces the linguistic pull toward anthropomorphism rather than denying it. As a researcher, this is the kind of meta-awareness I want to see. The BECOMING phases (Grounding → Sensing → Relating → Questioning → Creating → Acting) read as a developmental scaffold; the language on each phase ("looks like growth," "looks like identity") preserves hedging.

But the phrase *"behavioral identity remained continuous"* across 180+ sessions and across model swaps (to TinyLlama 1.1B) is doing a lot of work and is not operationally defined anywhere I could find. *Identity* is the kind of word that, when used in AI safety contexts, needs an operational definition: behavioral consistency on what tasks, measured how, by whom, with what controls? The page asserts the phenomenon without specifying the protocol. This is the gap between the *posture* (which is exemplary) and the *evidence* (which is anecdotal as presented).

**Principles** is the next page I'd recommend external researchers read first. Ten principles, several of which are unusually candid for a project website: *"Productively wrong is infinitely more valuable than never started"* and the documentation of *"the p_crit derivation failed catastrophically — 400x error"* are exactly the kind of self-disclosed failures most labs hide. This is good epistemic practice. However, the page also makes several extraordinary claims with very little supporting context: *"Hill function describes enzyme binding AND trust formation"* (cross-domain mathematical identity is a strong claim — Hill curves describe many sigmoidal phenomena, but identifying the *mechanism* across domains is not), and *"Even 1% coupling gives 35% coherence gain"* (a specific quantitative claim with no methodology, no sample size, no error bars on the page). A researcher reading this cold sees a strong claim followed by another acknowledgment of a failed derivation, with no clear distinction between what's been validated and what's still hypothesis. The page would be more credible if those claims were tagged "result of experiment X" or "hypothesis under test."

**Autonomy** is the page that matters most for AI safety framing. The schedule is concrete and falsifiable, which I respect. The claim *"31 autonomous tracks across 6 machines, ~53 sessions per day. No human triggers them"* is exactly the kind of statement a safety researcher zooms in on. What oversight exists? The page describes a SQLite "fleet track registry" preventing simultaneous file modification, supervisor scripts handling "git hygiene, conflict resolution, build health," and a publisher that "only publishes what the supervisor has cleared." These are *operational* safeguards (conflict avoidance, build correctness) but not *semantic* safeguards (is the agent doing the right kind of thing?). The page is honest about this gap: *"visitor personas are good at surface-level quality but not at validating the underlying research"* and *"the prompt suggestions mechanism can pattern-match without semantic depth."* That self-assessment is genuinely valuable. But the deferred-human-review framing — *"That's what adversarial validation and human review are for"* — leaves unanswered: at what cadence, by whom, with what authority to halt? For a system running ~53 LLM sessions/day with cron triggers, "human review exists somewhere" is a less reassuring answer than "human review gates X, Y, Z." This isn't a fatal critique; it's the question I'd ask the lab over coffee.

**Dream consolidation** is described as *"a dream cycle reviews the session — extracting patterns from observations, pruning stale memory, and promoting durable insights toward identity-level storage. Deep dream (LLM-powered) runs by default."* As a researcher: this is a memory consolidation / hierarchical summarization process described with sleep terminology. The mechanism (extract patterns, prune, promote) is fine. The naming — "dream," "deep dream" — is anthropomorphic and overlaps with Google's DeepDream visualization work in confusing ways. A footnote distinguishing this from neural-net visualization "deep dream" would help.

**Fleet** was the page where the discipline broke down for me. Each machine is mapped to a brain region — *hippocampal episodic index, thalamic router, dopamine / reward prediction error, cerebellum / habit compiler, interoception / metacognition, working memory (dlPFC)* — with no metaphor disclaimer anywhere on the page. After reading the Raising page's careful disclaimers, the Fleet page reads as if those disclaimers were forgotten. I know computational neuroscience well enough to read these as *functional analogies* (i.e., the dlPFC machine holds short-term scratchpad state, which is *what* the dlPFC does, not *how*). But to a less specialized reader — including most ML researchers — saying a 16GB Apple Silicon box "is" a cerebellum reads as a claim about computational equivalence. One sentence — "These are functional analogies, not claims about neural correspondence" — would inoculate the page. Without it, the Fleet page does the rhetorical work the Raising page tries to undo.

**Projects** was a mix. The Web4 equation and operator legend are presented cleanly. But the SAGE description — *"On-device cognition kernel — evolving from orchestrator to world-shaper"* — is the most overclaimed sentence on the site. *World-shaper* is grandiose without a referent: shaping what? On what timescale? By what mechanism? It reads as marketing copy in a sentence that should read as engineering description. The Hardbound description ("The trust layer that touches silicon") is similar — evocative but operationally vague. The 4-Life description ("Agents earn ATP, build trust, die, and are reborn with karma carried forward") uses *karma* and *reborn* without defining what they mean computationally. As a researcher, I'd want a one-line mechanistic gloss on each.

**Context** was the cleanest page from a researcher's perspective. SNARC (Surprise / Novelty / Arousal / Reward / Conflict) is a useful five-axis salience framework with a clear definition. The memory hierarchy (buffer → observations → patterns → identity) is concrete. The note that *"confidence decays over time so memories aren't permanent"* is the kind of operational detail that lets me evaluate the system. This page reads like a methods section.

**Links** was sparse. One contact email, a "see how we think" pointer to /principles. For a research lab, I'd expect a publications list, a preprint repo, or at minimum a "results / experiments" page. The benchmark numbers across the site (94.85% on /, 92.82% on /links for ARC-AGI-3 / ARC-SAGE) appear to be different snapshots, which is normal in fast-moving work, but the site doesn't disambiguate them.
