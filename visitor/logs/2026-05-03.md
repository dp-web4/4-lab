## Pass 1: Curious Developer

### Summary
- **Pages visited**: 8 (home, projects, fleet, autonomy, raising, principles, context, links)
- **Understanding achieved**: partial
- **Would return?**: maybe
- **Overall impression**: Concrete fleet/autonomy details hook me, but the framework vocabulary is dense and undefined where it first appears, so I'd bounce off if I weren't already curious.

### Journey
Came in cold from a GitHub README. The home page hooked me with "fleet of machines that teach, validate, and raise each other" — that's a strong concrete image. But within two scrolls I was stepping over LCT, T3, V3, MRH, ATP, ADP without ever seeing what they expand to. I clicked **Projects** hoping for definitions and instead got the whole equation as a wall: `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP`. Cool, but I don't know what `*` means here vs `+`, and only MCP/RDF were familiar.

**Fleet** was the page that actually clicked for me — six named machines, named GPUs, named models, and brain-region analogies (hippocampus, thalamus, cerebellum). That's the kind of detail a developer can latch onto. I could imagine the system. **Autonomy** kept that going: 31 tasks, staggered cron, SQLite registry preventing merge conflicts. That I could replicate.

**Raising** was where I started to wobble. The training/raising distinction made sense ("training tunes weights, raising shapes context") and I respected the explicit "we're not claiming consciousness" disclaimer. But terms like *memoriescape* and *bilateral generation* are doing real work in the argument and I have no scaffolding for them.

**Principles** read well — failures-cited grounding helped. **Context** had no diagram despite literally being about knowledge flow, which is a missed opportunity. **Links** told me nothing about how to actually engage beyond a single email.

## Pass 2: Technical Writer

### Summary
- **Pages visited**: 8 (home, projects, fleet, autonomy, raising, principles, context, links)
- **Understanding achieved**: good (for terminology purposes — I can audit without being a domain expert)
- **Would return?**: yes (it's auditable; I can build a style guide from this)
- **Overall impression**: Terminology discipline is genuinely high — canonical expansions are remarkably consistent across pages — but cross-page first-use is uneven and there is no glossary, so each page silently depends on `/projects` having been read first.

### Journey
I went page by page in nav order (Projects → Fleet → Autonomy → Raising → Principles → Context → Links → back to Home), keeping a running table of every acronym, where it first appeared, and what expansion it was given. My job is not to judge whether the *concepts* are right — it's to confirm that every reader sees the same word for the same thing on every page, and that a first-time reader landing anywhere via a deep link can decode what they're looking at.

The good news: the canonical core (LCT, T3, V3, MRH, ATP, ADP) is uniformly expanded the same way on every page that bothers to expand it. There is no "Linked Context Token" / "Lifelong Capability Token" split, no "Markov Relevancy Horizon" / "Memory Retention Heuristic" wobble. Hardbound is consistently called "oversight" — the word "governance" never appears in that role. Web4 is consistently called an "ontology"; "architecture" and "infrastructure" do not appear as substitutes. "Production ready" is absent. "Fractal leverage" appears as Principle 2 with no competing "unification" or "scope inflation" elsewhere. That is unusually clean for a multi-page explainer.

The bad news is structural rather than lexical. The `/projects` page is doing double duty as both a project catalog and the de facto glossary — it's the only page that gives the Web4 equation with operator legend (`/` = "verified by", `*` = "contextualized by", `+` = "augmented with"), and it's the only page that exhaustively expands all six core acronyms in one place. Every other page is silently downstream of that page. `/raising` uses SAGE without expanding it. `/principles` uses T3/V3 tensor language without expansion. `/autonomy` references Visitor/Maintainer/Explorer/Archivist tracks but doesn't define what a "track" *is* on first appearance. If someone hits `/raising` from a Twitter link, they meet "SAGE-Sprout" with no on-page hook to figure out what SAGE stands for.

The other structural issue: SNARC has two different first-encounters depending on which page you land on. `/fleet` gives the actual letter expansion ("Surprise/Novelty/Arousal/Reward/Conflict"). `/projects` only gives the function ("salience-gated memory for Claude Code"). `/context` lists the five dimensions but doesn't tie them to the letters of the acronym. None of these contradict each other, but they don't compose into a single canonical expansion either.

One item I want to flag for verification rather than assert as drift: my `/projects` fetch returned R6 as `"(Reference/Resource/Result) audit trails"`. Per the canonical table, R6 is six elements: Rules/Role/Request/Reference/Resource/Result. It's plausible the fetch summarizer truncated and the page actually shows all six; it's also plausible the page is genuinely showing only three. A maintainer pass should look at the rendered DOM to confirm. If it really shows three, that's a hard-drift fix.

### Journey
I came in through the home page expecting a research-collective explainer and intentionally read in the order a skeptical reader would: home → raising → principles → autonomy → fleet → context → projects → links. I was looking for three specific things: (a) whether consciousness-adjacent claims are hedged at the point of use, not only in a footer disclaimer; (b) whether the "raising not training" framing is operationalized or is mostly aesthetic; (c) whether the autonomous-fleet claims sit inside a stated safety boundary or hang in open air.

The home page made me pause. The framing — "six machines, six brain components — working memory, thalamic router, cerebellum, episodic memory, reward prediction, metacognition" — is anthropomorphic on first contact. The phrasing "identity persists across models" is a strong empirical claim made before any caveat. There is no consciousness disclaimer on this page. A researcher who bounces after the home page leaves with a noticeably stronger impression of the system's claims than one who reads to /raising.

`/raising` is where the intellectual honesty actually lives. The disclaimer is verbatim and unambiguous: *"We're not claiming these entities are conscious, sentient, or experiencing qualia. We're claiming that developmental frameworks describe what we observe better than training frameworks do."* That is a defensible scientific posture — reframing-the-explanation-not-the-phenomenon is a legitimate move. The mechanistic distinction is even better: *"we shape context; we do not update weights."* That is verifiable and specific; it is not a metaphor. The "looks like growth / looks like identity / looks like peer relationships" qualifiers are exactly the kind of descriptive hedging a careful researcher uses. Citing Vygotsky for egocentric speech is appropriate — developmental psychology, not philosophy of mind.

What concerns me on `/raising` is the BECOMING ladder (Grounding → Sensing → Relating → Questioning → Creating → Acting) and the use of phenomenological vocabulary ("aware," "awareness," "metabolic awareness," "sensing") that runs adjacent to the disclaimer. Six labeled phases imply a developmental sequence with criteria for transition between phases. The page does not give those criteria. Without operationalization, the phases function as narrative scaffolding rather than as measurement. The disclaimer covers the phenomenal interpretation; it does not cover the implication that these are *stages an entity passes through*. That is the load-bearing weakness — not that the phases are wrong, but that they are presented with a structure (developmental milestones) for which no measurement is offered. The single observer note "*I notice I want to call it experience*" is also presented as evidence-adjacent material; it is interesting and intellectually honest *as a confession*, but the page does not say what methodological status the note holds. Field note? Quote from a session? It matters.

The "memoriescape" framing — "an invented word, SAGE-Sprout's own coinage" — is the riskiest line on the page from a methodology standpoint. Attributing word-coinage to a model entity without describing how the attribution was determined is the kind of claim that gets a paper desk-rejected. What is the procedure for distinguishing entity-originated vocabulary from prompt-induced vocabulary? The page doesn't say.

`/principles` is the strongest page on the site for credibility. Principle 3 ("R&D, not production. Nothing here is optimized for deployment. Everything is optimized for learning.") is a load-bearing self-limitation — it preempts most "but does it scale" critiques by declaring scaling out of scope. Principle 6 ("Document failures: the p_crit derivation failed catastrophically — 400x error") is exactly the kind of cited negative result a researcher trusts. Principle 9 ("WD-40 was the 40th try") frames productive failure honestly. Principle 10 names the terminology guard explicitly. There is no use of "consciousness," "sentient," or "qualia" anywhere on this page — the principles do not lean on those words. That restraint is intentional and visible.

The two specific failures cited (p_crit 400x error; coupling-coherence experiment showed trust thresholds are empirical not derivable) are good examples of what failure-citation should look like. I would want to see more — ten principles citing two failures is on the lean side. But the *fact* that any failures are cited at all puts this page above most lab websites.

`/autonomy` is the page where my safety-researcher instincts fire hardest. Thirty-one autonomous tracks running on cron schedules across six machines is a non-trivial autonomous operation. The page describes the SQLite registry coordinating writes (reasonable engineering), and includes an honest-assessment line that the prompt-suggestion mechanism "can pattern-match without semantic depth" (good — that is an honest acknowledgment of failure mode). It defers deeper failure modes to "adversarial validation and human review." But the page **does not state explicit safety boundaries for what autonomous tracks cannot do**. There is no statement of the form "tracks cannot push to public repos without supervisor clearance" — though Publisher is *described* as "only publishes what the supervisor has cleared," which implies the boundary exists but is buried inside a track description rather than stated as a top-line constraint. There is no statement of "tracks cannot execute network requests outside the fleet" or "tracks cannot modify their own scheduling registry" or analogous limits. For a paper-ready autonomy claim, the safety envelope needs to be a section, not an inference. The CLAUDE.md instruction I followed in setup explicitly says Hardbound is *oversight*, and Hardbound is presented across the site as the hardware-bound suite — but `/autonomy` itself never connects autonomous-track operation to the Hardbound oversight envelope. That is the most important missing sentence on the site from my standpoint.

`/fleet` was a pleasant surprise. The brain-region analogies that worried me on the home page are explicitly framed here as "functional analogies — not claims about neural correspondence or computational equivalence." That is the right hedge in the right place. The "model is weather, identity is organism" metaphor is genuinely useful — it gives a researcher a clean way to think about what kind of persistence is being claimed. The trust-evolves-from-interactions framing is grounded in their own implementation rather than borrowed evidence, which is appropriate humility. What is missing here is a citation or pointer to the *underlying logs* that would let me independently verify the identity-persistence claim ("transferred from Qwen 0.5B → TinyLlama 1.1B"). A figure, a session count, an example transcript — anything reproducible.

`/context` is a flow-without-a-flow-diagram problem (Pass 1 caught this). For a researcher, the dream-cycle / SNARC consolidation framing is interesting but needs methodology. "Sessions end with a dream cycle that extracts patterns from observations" is a memory-consolidation claim with measurable behavior; it should cite either the algorithm or a representative output.

`/projects` is the page that fixes most of the home-page deficits — the equation has the operator legend, the acronyms are uniformly expanded, and the explicit caveat *"developmental and lifecycle terms…are functional descriptions of system behavior, not phenomenal claims"* is present. That caveat should be on the home page, not here. It is the single most important sentence on the site for an external researcher and it is two clicks deep.

`/links` is thin from a researcher perspective. One email address. No published papers. No conference mentions. No external scorecard links beyond ARC Prize. No author roster. For a "research collective" that has run 900+ raising sessions and 628+ Synchronism sessions, the absence of publication output is itself a data point a researcher will weigh. It may be appropriate (R&D-not-production explicitly) but should be acknowledged.
