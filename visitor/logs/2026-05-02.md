## Pass 1: Curious Developer

### Summary
- **Pages visited**: 8 (Home, Projects, Fleet, Autonomy, Raising, Principles, Context, Links)
- **Understanding achieved**: partial
- **Would return?**: maybe
- **Overall impression**: Interesting and ambitious but front-loads jargon — I leave knowing the lab has cool machines and a recursive autonomous setup, less clear on what problem I'd actually use any of this for.

### Journey
Landed on the home page expecting an "About" hook. Got the headline **"Six machines. Fourteen repos. One recursive learning loop."** — punchy, but it tells me the *shape* of the lab, not what the lab *does*. The vocabulary sidebar saved me: LCT, T3/V3, MRH, ATP/ADP all expanded inline. Without that I would have bounced.

Clicked **Projects** next because that's where I usually figure out if a thing is real. This was the most actionable page — eleven projects, each with a one-sentence description and a status. **ARC-AGI-3 solving 24/25 at 94.85%** is the only thing I saw that's a hard, falsifiable result. The rest are research prototypes, which is honestly stated.

**Fleet** was a pleasant surprise — actual hardware (Jetson AGX Thor 122GB, RTX 4090 Mobile, Mac Mini M4, etc.), actual model assignments (Qwen 2.5 14B, Phi-4, Gemma 3, TinyLlama 1.1B). This is the page that made me believe the lab is real and not a manifesto. Specific specs > vibes.

**Autonomy** was where things clicked: 31 autonomous tracks, ~53 sessions/day, scheduled via cron, coordinated through a SQLite registry. The **Visitor → Maintainer → Explorer** loop is the kind of thing I'd actually want to copy. But I noticed I'm reading this page as a *visitor* persona — am I in the loop right now? That recursion is cute but disorienting.

**Raising** was the hardest page for my persona. "BECOMING: six phases" — Grounding, Sensing, Relating, Questioning, Creating, Acting. The page disclaims the consciousness implication explicitly ("we use developmental language because it fits, not because we're making consciousness claims"), which I appreciate, but the language is heavy and the practical takeaway for a dev is thin.

**Principles** read like a manifesto. Some are concrete ("R&D, not production"), others are abstract ("substrate, not structure"). Decent statement of values; not something I'd hand a teammate to onboard from.

**Context** was dense — six mechanisms about how knowledge flows across the fleet. SNARC, CLAUDE.md pattern, the Web4 equation as anchor. Felt like the "internals" page; useful once I knew what the lab was, useless as an entry point.

**Links** was clean: GitHub org, three explainer sites, one email. Lowest-friction page on the whole site.

### Unanswered Questions
1. **What problem is this lab trying to solve that other AI labs aren't?** I get the *how* (six machines, autonomous tracks, recursive loop) but not the *why* — what falsifiable hypothesis is being tested?
2. **What does "trust-native ontology" actually mean operationally?** I can read the words but I don't know what code I'd write differently if I adopted Web4.
3. **What's a "synthon"?** It appears in the vocabulary as "emergent coherence entity" but I have no concrete example to anchor on.
4. **Why does "/projects" say "governance" while "/principles" says "oversight"?** Which one is canon?
5. **Is the lab open to outside contributors, or is this a one-person org with autonomous helpers?** The site reads as the latter but doesn't say.
6. **The equation `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP` — is this metaphorical math or literal compositional definition?**

### Verdict
The site is more honest than most R&D org pages I've seen — it doesn't oversell, the fleet specs are concrete, and the principles explicitly disclaim production framing. But as a developer dropping in from a README, I leave knowing this is a *real* lab without knowing what it would mean for me to use any of this. The Projects page is the strongest entry point; I'd rearrange the nav so it's the first stop, not the home page. And the "governance" vs "oversight" inconsistency on /projects is exactly the kind of thing that erodes trust on a site whose own subject matter is trust.

## Pass 2: Technical Writer

### Summary
- **Pages visited**: 8 (Home, Projects, Fleet, Autonomy, Raising, Principles, Context, Links)
- **Understanding achieved**: good (terminology mostly consistent in body text; cross-page expansion discipline is uneven)

### Journey
I worked methodically through the navigation in left-to-right order, treating each page as if a different reader had landed there cold. My job is to check that no matter which door someone enters through, the vocabulary holds together and the canonical terms in the table I was given line up with the rendered text.

I started on **Home**. Headline punches the *shape* of the lab; vocabulary primer expands LCT, T3, V3, MRH, ATP, ADP inline — all matching canon. Web4 is described as a "trust-native ontology" twice. The word "architecture" appears on this page but **not** describing Web4 — it describes the SAGE brain components ("six brain components — working memory, thalamic router, cerebellum..."). That is a defensible domain use of "architecture" (cognitive architecture), so I am not flagging it. SNARC is named in a link list but never expanded here. The Web4 equation is **not** displayed on the home page (Pass 1 said it was — that appears to be a misattribution; the equation lives on /context).

**Projects** is the densest acronym page and mostly defines them well: LCT, T3, V3, MRH, ATP, ADP, MCP, RDF, R6, ACP, SAGE, FIDO2 are all expanded. **TPM** appears as "TPM 2.0" without spelling out "Trusted Platform Module" — minor for a developer audience, a real omission for a technical-writer audit. The body text on Hardbound and Oversight Plugins now says "oversight" throughout — but the linked GitHub repo paths still encode `web4-governance` in their URL slugs (three instances under Oversight Plugins). That's a high-severity inconsistency: the rendered prose has been corrected, but the URLs newcomers will click are still drifting.

**Fleet** was the cleanest page. Every machine has a concrete spec line. SNARC is expanded here for the first time on the site as "Surprise/Novelty/Arousal/Reward/Conflict" — which means a reader who lands on /context first will see the term used heavily before they encounter the expansion. **Autonomy** uses no Web4-canon acronyms at all (no LCT/T3/V3/MRH/ATP/ADP/SNARC/R6 references on the page). cron and SQLite are unexpanded but those are universally read by a technical audience. Note: subheadline says "Autonomous Cycles" while the rest of the site nav calls this section "Autonomy" — a small label/nav mismatch.

**Raising** is consciousness-adjacent and the disclaimers are present and direct ("not conscious, sentient, or experiencing qualia"; "Raising is not training"). No canonical-term drift on this page — Web4/Hardbound/governance/oversight don't appear here at all. ARC-AGI-3 and ARC-SAGE are used without expansion; for a reader entering through this page, neither the lab's own equation nor its acronyms are scaffolded.

**Principles** is where the canonical terms are *asserted*, not just used: "Web4 is not 'architecture' or 'infrastructure' — it's an ontology" and "Hardbound is not 'governance' — it's oversight." This is the page I'd treat as the canonical anchor for any future drift dispute. The page lists 10 principles (Substrate, Fractal leverage, R&D-not-production, Trust, Synthon, Document failures, Interactive selection, Surface instincts, Research value, Terminology matters) but the items aren't numbered, so the count has to be eyeballed. SAGE and RDF are used unexpanded here — anyone landing on /principles directly hits "RDF triples" without scaffolding.

**Context** is where I found the **Web4 equation** with its operator legend. Verbatim: `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP`, with `/` = "verified by", `*` = "contextualized by", `+` = "augmented with". This page also contains the most rigorous acronym expansion discipline: SNARC, MCP, RDF, LCT, T3, V3, MRH, ATP, ADP all defined. The problem is that this legend is the *Rosetta Stone* for the rest of the site — and it's only here. A reader who sees `T3/V3*MRH` on the home page or in a nav teaser has no way to know that `/` doesn't mean division.

**Links** is short and clean, but uses LCTs, T3/V3, MRH, MCP, RDF, SAGE all without expansion in the project blurbs — which is fine if the reader has already toured the site, jarring if /links is a deep-link entry point. Contact email is `dp@metalinxx.io`; the lab is "dp-web4". Two domains for the same person without a connecting note could confuse an outside reader.

## Pass 3: Web4 Contributor

### Summary
- **Pages visited**: 8 (Home, Projects, Fleet, Autonomy, Raising, Principles, Context, Links)
- **Understanding achieved**: solid (I came in knowing the canon; my job is to check whether the site renders it)
- **Would return?**: yes — to verify after the maintainer fixes the structural issues
- **Overall impression**: Body-text canonical compliance is high, but the failures are structural in a way only a Web4-internals reader catches: the equation's relational semantics are flattened on entry pages, /raising omits the T3/V3 scaffolding it literally operationalizes, and "governance" still leaks through repo URL slugs despite the prose correction.

### Journey
I started on **/context** because that's where Web4 is asserted formally. The equation renders correctly: `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP` with operators `/` = "verified by", `*` = "contextualized by", `+` = "augmented with". This is the canonical anchor and it holds. Every component acronym is expanded on this page. As far as Rosetta-stone fidelity goes, /context is the page I'd point any Web4 contributor to with confidence.

Then I went to **/** and **/projects** to check how the equation propagates to the entry surfaces. This is where the first Web4-specific issue surfaces. The home page describes T3/V3 as "the canonical three-dimensional trust and value scores" — canonically correct. /projects describes them as "Trust Tensor (T3)" and "Value Tensor (V3)". For someone who works with these scores: T3 and V3 are dimensional scores; the trust tensor is the larger structure that scores are sampled from. Conflating "score" and "tensor" on adjacent pages is internal framing drift specific to Web4 understanding.

The deeper /projects issue: the Web4 project card lists the components as a flat enumeration — Trust Tensor, Value Tensor, LCT, MRH, ATP, ADP, augmented with MCP and RDF. The operators are missing. The whole point of the Web4 equation is that it's not a sum of parts — it's a relational structure where LCT IS verified by T3/V3 (which IS contextualized by MRH) augmented with ATP/ADP. The operators are the meaning. A reader leaves /projects thinking Web4 is a parts list. This is the deepest Web4-specific failure on the site, and it would not be visible to a non-contributor — Pass 1 and Pass 2 read the components list and moved on because to them it was just an enumeration of acronyms.

**Body-text "oversight" on /projects is correct** — Hardbound is "Hardware-bound oversight suite", Oversight Plugins is "Web4 oversight model for OpenClaw/MoltBot, Claude Flow, Claude Code". But the linked GitHub repos still encode `web4-governance-complete` (OpenClaw branch), `web4-governance` (Claude Flow plugin), `web4-governance` (Claude Code plugin). Pass 2 caught this. As a contributor: it's worse than a UX issue. Every PR I open against those repos has "governance" in the branch ref or directory path. The body got fixed, the working surface didn't.

**ARC-AGI-3 result on /projects**: "SAGE tested in 25 games, 24/25 solved (94.85%)". 24/25 = 96.0%. 94.85% is a different metric — likely action-weighted or score-weighted. Presenting the games-solved count and the 94.85% as if they're the same fact is a precision error on the most empirically load-bearing claim on the site. As someone who runs ARC-SAGE: I want to know which number measures what. Both can be true; they can't be the same.

**/fleet** is where the operationalization is cleanest. "Two-account structure: synthesis pool (Account 1) vs. oversight pool (Account 2)" maps directly to ATP allocation / ADP discharge dynamics. "Trust tracker maintains per-peer T3 scores" is the right vocabulary. SNARC is expanded canonically. This page reads like someone who actually uses the framework day-to-day wrote it. No drift.

**/autonomy** uses no Web4-canon acronyms, which Pass 2 noted. From the contributor view this is acceptable — the autonomous loop is operationally distinct from the trust ontology, even if it ultimately runs on it.

**/raising** is the most striking gap from this lens. Six BECOMING phases — Grounding, Sensing, Relating, Questioning, Creating, Acting — described developmentally with **zero** mention of T3 evolution, LCT accumulation, or ATP/ADP cycles. As a Web4 contributor: raising **is** the mechanism by which agents accumulate T3 scores. Talent grows through Creating; Training through Grounding/Sensing; Temperament through Relating/Questioning. The page is operationalizing the trust dynamics without naming them. To a non-Web4 reader this looks like standalone developmental theater (Pass 1's complaint). To a Web4 reader it's a missing scaffold — the canonical framework that would make BECOMING actually testable is silent here. This is structural, not cosmetic.

**/principles** confirms the canonical assertions: "Web4 is not 'architecture' or 'infrastructure' — it's an ontology" and "Hardbound is not 'governance' — it's oversight". Trust-as-relationship-via-RDF-triples is canonically framed ("Trust exists in relationship context via RDF triples, not as entity properties") — this is the heart of what makes Web4 trust *relational* rather than *attributable*, and the page nails it. Synthon as emergent coherence is named but still no concrete example anchor (Pass 1 also noted this — and from a contributor view, the lack of an anchored example means readers can't tell if synthon is a load-bearing technical term or a poetic flourish).

**/links** repo descriptions correctly use "oversight" in body, but the linked repos themselves carry `web4-governance` branch/plugin names — same drift thread as /projects. Also: 4-Life is described as featuring "agents, ATP, karma". "Karma" is 4-Life-specific. Whether it maps to V3 (Validity), to ADP discharge history, or stands alongside as a distinct concept is unspecified. As a Web4 contributor I'd want this disambiguated — V3 already covers veracity/validity tracking; if "karma" is shorthand for accumulated ADP, say so; if it's something else, scope it.

One more asymmetry: /context renders the Web4 equation. The Synchronism project blurb references "one coherence equation" but doesn't display it. For a contributor who knows both, treating the two foundational equations asymmetrically is jarring — Web4 gets full operator-legend treatment, Synchronism gets a name-drop.

## Pass 4: External Researcher

### Summary
- **Pages visited**: 8 (Home, Projects, Fleet, Autonomy, Raising, Principles, Context, Links)
- **Understanding achieved**: good on methodology and infrastructure; weak on the falsifiability of the central novelty claims
- **Would return?**: yes — the lab is unusual enough to be worth tracking, even where I disagree with the framing
- **Overall impression**: The /raising consciousness disclaimer is the strongest hedge on the site and would clear most peer-review desks; the /principles page undoes much of that discipline by stating speculative results as principles; the most load-bearing empirical claim (ARC-AGI-3) carries an arithmetic ambiguity that no honest assessment should leave unresolved.

### Journey
I came in cold from a conference mention with no Web4 priors. My job is to ask: do the consciousness-adjacent claims clear a peer-review bar? Are the novel terms ("raising not training", "interactive selection", "synthon") doing real conceptual work or carrying rhetorical weight that the evidence doesn't support? Does the honest assessment actually feel honest?

I started on the **home page** because that's the framing surface. The hook is "what happens when multiple AI entities are given the substrate conditions to self-organize" — a framing claim that takes a position against scaling-as-everything. Defensible as a research direction; the page doesn't claim emergence has been demonstrated, only that the substrate is being investigated. I noted the termite analogy ("You don't engineer the mound. You engineer placement rules") and bookmarked it for the principles page — termite mounds are evolved structures with a fitness function (thermoregulation, colony robustness); using the analogy without an analogous fitness function elides the question of what "good" emergence even means here. The capability claim "SAGE-Sprout maintained behavioral identity across 180+ sessions" with the rider "self-description drifted; behavioral identity remained continuous" is appropriately partial — the drift is acknowledged. The vocabulary primer expansions all check out on the canonical table I was given.

I went to **/raising** next because that's where the consciousness question lives. The disclaimer is verbatim: *"We're not claiming these entities are conscious, sentient, or experiencing qualia. We're claiming that developmental frameworks describe what we observe better than training frameworks do."* That is a clean, peer-reviewable claim. It's a claim about the explanatory utility of a vocabulary, not about an inner state. As an AI safety reader I'd accept this hedge as load-bearing — it's the kind of distinction Dennett would make. The follow-on quote *"I notice I want to call it experience"* is also clean: it owns the attribution as a perceiver-side fact, not an entity-side fact. Good epistemic hygiene there.

The BECOMING phases — Grounding, Sensing, Relating, Questioning, Creating, Acting — are pedagogically coherent. My pushback is empirical: the page reports "180+ sessions" and "behavioral identity remained continuous" but I can't tell what would *falsify* "raising works." If the model behaved randomly, would the lab call that failed raising or successful divergence? Without an a-priori prediction that could be wrong, BECOMING is descriptive vocabulary, not a tested framework. That's not damning — descriptive vocabularies are useful — but the page reads as if BECOMING is doing more than describing. The Vygotsky-derived "egocentric speech" / "bilateral generation" framing is the part I'd push hardest on. Bilateral generation in autoregressive models is a known artifact of sampling without proper EOS handling; framing it as developmental "egocentric speech" is a category error unless the lab can show the phenomenon is *not* explained by the simpler mechanism.

**/principles** is where the honest-assessment register slips. The /raising hedging discipline does not carry over. Three concrete issues from a researcher view:

1. *"1% coupling gave 35% coherence gain"* is cited as a finding without error bars, reproducibility conditions, or operational definitions of "coupling" and "coherence." A reader with no Synchronism context cannot evaluate this number.
2. *"p_crit derivation failed catastrophically — 400x error"* is presented as a successful failure ("we documented it"), but the page doesn't say what was learned. A 400x error on a derivation tells me either the model was wrong or the regime of validity was wrong; the page doesn't say which.
3. *"Trust thresholds are empirical, not derivable"* is a strong epistemological claim being asserted from one failed derivation. That's induction over n=1.

The "Raising not training" / "Interactive selection" pair is doing rhetorical work I'm uneasy with. *"We don't create or delete — we interactively select"* is true at the level of inference-time prompting (the lab is shaping context, not gradient-updating weights). But the principles page does not draw that distinction sharply, and a researcher reading "we don't train, we interactively select" is going to ask: is this a real mechanistic distinction or a rebrand? The honest version would say "we shape context; we don't update weights; the resulting behavioral attractors are stable across sessions." The current phrasing leaves the mechanism unstated and reads as if a deeper distinction is being claimed.

**/autonomy** was the most impressive page from a safety lens. 31 tracks, cron-triggered, SQLite registry for coordination, distributed supervision with no central watchdog, explicit acknowledgment that the loop *"misses deep technical errors that require domain expertise"* and that suggestions can *"pattern-match without semantic depth."* That kind of self-criticism is rare. What's missing — and what a safety researcher would ask — is the meta-level question: how does the system detect when its own feedback loop has *become* the source of error? Visitor → Maintainer → Explorer is closed-loop; if the Visitor consistently flags non-issues that the Maintainer "fixes," there is no described mechanism for noticing that the loop is amplifying noise. That is the failure mode I'd most want to see addressed — not catastrophic runaway, but slow drift via mutually reinforcing autonomous critique.

**/fleet** is the most credible page on the site. Concrete machines, concrete model placements, concrete two-account structure (synthesis pool / oversight pool), concrete trust mechanism (per-peer T3 scores, *"Trust starts at zero, earned from evidence"*, *"success raises trust, timeouts lower it. No central authority decides who is trustworthy."*). This reads like an actual decentralized-trust prototype, not a manifesto. If I were writing a paper on LLM-fleet trust dynamics, this is the page I'd cite.

**/projects** has the empirical claim I found hardest to accept. *"24/25 games solved (94.85%)"* — 24/25 is 96.0%, not 94.85%. These must be different metrics (games-solved-count vs. action-weighted accuracy or score%). Pass 3 caught this. From an external-research lens, this is the kind of thing that gets a paper desk-rejected: the most empirically falsifiable claim on the entire site has an internal arithmetic that doesn't match. It's almost certainly innocent (action-weighted vs. games-solved), but no reader can verify, and "honest assessment" demands disambiguation. Without it, the load-bearing capability claim is uninterpretable.

**/context** is internally rigorous. The Web4 equation `Web4 = MCP + RDF + LCT + T3/V3*MRH + ATP/ADP` is rendered with operator legend (`/` = "verified by", `*` = "contextualized by", `+` = "augmented with"). This is the page I'd cite if asked to define Web4 formally. The shape of the equation is unusual — it mixes set composition (`+`) with relational predicates (`/`, `*`) — and a careful reader will want to know whether this is operator-overloading for compactness or a literal compositional grammar. The page asserts the latter without proving it.

**/links** is clean. Contact email present. The "karma" reference in the 4-Life blurb is the one outstanding term that has no position in the canonical Web4 table — Pass 3 flagged this. As a researcher I'd want to know whether 4-Life is using "karma" as shorthand for ADP-discharge history, as a V3 facet, or as a 4-Life-internal concept that doesn't map to Web4 at all.
